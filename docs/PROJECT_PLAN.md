### 1.2 架构现状说明

项目当前运行模式为**单体模式**（`cmd/server` 单入口），但内部架构完全按照**微服务标准**设计和实现：

- **服务分层**: `internal/svc-user/`, `internal/svc-media/` 等独立服务目录
- **接口定义**: 使用 proto 文件定义服务接口，位于 `api/proto/v1/`
- **依赖注入**: 支持 Wire DI，各服务可独立组装
- **消息队列**: Watermill GoChannel 作为进程内 PubSub，未来可替换为 RabbitMQ/Kafka
- **数据层**: Ent ORM 统一实体定义（`internal/data/entity/`）

**架构决策**: 单体模式便于当前快速开发和测试，微服务架构设计确保未来可按服务边界平滑拆分。

### 2.1 关键脱节点

| 问题 | 文档状态 | 代码状态 | 严重程度 |
|------|---------|---------|---------|
| ORM技术栈 | DEVELOPMENT_PLAN.md 写 GORM | 实际使用 Ent | 🔴 高 |
| API路径 | 部分文档写 `/media` 单数 | 后端实现 `/medias` 复数 | 🔴 高 |
| 响应格式 | 前端期望 `{list, total}` | 后端返回 `{code, message, data: {items, total}}` | 🔴 高 |
| HTTP方法 | 前端使用 PUT | 后端使用 POST（通知已读） | 🔴 高 |

### 2.2 根本原因

1. **迁移半途而废**: 前后端各自修改，未做联调对齐
2. **文档滞后**: 开发速度超过文档更新速度
3. **API版本混乱**: 旧handler未清理，新路由已启用，前端调用路径混乱
4. **测试覆盖不足**: 虽有集成测试，但前后端联调测试缺失
